OAuth2.0 Components

Functional Features of OAuth2.0

  • It is not required for Clients to support password authentication or store user credentials.

  • Clients obtain authorized access, i.e., particular access to user-authenticated services.

  • Users can revoke delegated access by the client at any time.

  • OAuth2.0 access tokens will get expired after a set validity period.

  • If the client encounters a security breach, user data will get compromised only until the validity period of the access token.

OAuth2.0 Working Principle

OAuth Authentication

The OAuth authentication involves the following steps;

  • Verifying the Application

  • Generating Tokens

  • Refreshing the Access tokens

  • Revoking Tokens

Explanation

First, it is necessary to register the admin user details on the OAuth server and then register the application on the Developer Portal. And, then this generated Client ID, and the Client Secret should be configured with the specific application.

After successful integration, the application will request for an access token, where the OAuth server creates a token, which will act as a bridge between the application and the server. So, whenever the end-user tries to log in to the company, the application redirects the user to the server login page.

And, once the user provides the login credentials, the server authenticates the user and issues a token which will be manages by the application. If the user is a registered user, the server logs in to the specified company, else the session gets expired.

Prerequisites

  • The user must have a valid Account with OAuth server.

  • The application must have permission to access API service (System-level security settings)

  • The user must have the required role to access the ZKTeco’s Authorization Server API

  • Active Internet connection to send requests and get responses.

  • A revoking token application leads to the withdrawal of permissions from the client application linked to that token to access the secure resources of the end-user.

Authorization Code Grant Type

The Authorization Code or Web server flow is suitable for clients to interact with the end-user user-agent (typically a Web browser), which receives the incoming requests from the authorization server (which acts as an HTTP server).

If you are building a server-side (or "web") application that is capable of storing secrets, securely then the authorization code flow is the recommended method for controlling access to it.

Request Path

http://serverIp:serverPort/oauth/authorize?response_type=code&state={state_code}&client_id={client_id}&redirect_uri={redirect_url}

Request Method

GET

Content Type

application/json

Parameter Description

state_code

  • The state parameter serves two functions.

  • When the user is redirected back to your app, whatever value you include as the state will also be included in the redirect.

  • This gives your app a chance to persist data between the user being directed to the authorization server and back again, such as using the state parameter as a session key.

  • This may be used to indicate what action in the app to perform after authorization is complete, for example, indicating which of your app’s pages to redirect to after authorization.

client_id

  • The client_id is the identifier for your app.

  • You will have received a client_id when first registering your app with the service.

redirect_url

  • The redirect_uri may be optional depending on the API but is highly recommended.

  • This is the URL to which you want the user to be redirected after the authorization is complete.

  • This must match the redirect URL that you have previously registered with the service.

response_type

  • response_type is set to code indicating that you want an authorization code as the response.

Implicit Grant Type

The Implicit Grant (User-Agent) authentication flow is for the client applications (consumers) residing in the user's device. This flow may be implemented in a browser using a scripting language such as JavaScript, or from a mobile device or a desktop application as these consumers cannot keep the client secret confidential (application password or private key).

Use the implicit flow for token acquisition if you are planning a SPA architecture that has no backend components or if you intend to invoke a web API via JavaScript.

This Implicit Grant is for the user-agent type of clients (e.g., single-page web apps), which does not store a client secret, as all the application code and the storage is easily accessible.

Password Grant Type

The Password grant is one of the smooth OAuth grants and involves the following process. The application navigates the user to a login interface that requests for the username and password to collect the user credentials and then executes a POST request to the server to exchange the password for an access token.

The advantage of this grant type is that it lets the application take advantage of the rest of the benefits that OAuth provides around access tokens and token lifetimes. Instead of storing the user’s password on the device, the application only has to touch the user’s password for as long as it takes to get the access token, then it can store and use the access token instead.

This grant is a great user experience for trusted first-party clients both on the web and in native device applications.

Glosary: Terminologies

Client

An application that sends requests to the resource server to access the protected resources on behalf of the end-user.

Client ID

 The consumer key generated from the connected application.
Client Secret The consumer secret generated from the connected application.

Authentication server

 Authorization server provides the necessary credentials (such as Access and Refresh tokens) to the client.

Authentication code

 The authorization server creates a temporary token and sends it to the client via the browser. The client will send this code to the authorization server to obtain access and refresh tokens.

Access Token

 A token that is sent to the resource server to access the protected resources of the user.

Refresh Token

A token that can be used to obtain new access tokens.
